Implementing SOLID UI Components using Go

Jun 7, 2017 #UX #Design

Introduction

SOLID principles have been applied to OOP software design in order to make software less rigid, less fragile, reusable, and more natural. In a previous blog post I talked about applying SOLID principles to Web components. I gave a very high overview of a client side implementation of SOLID components using an HTML5 technology called Web Components. It turns out that with practiced discipline existing frameworks can support a SOLID approach.

In this post I will apply the same principles on the server side by showing how the Go templating language can be used to support the definition and composition of components.

Problem Setting

Few of the benefits of SOLID design are visible in a trivial system. Many critics of SOLID principles claim that the approaches used to implement them are over-engineered. That is true of the simple examples used to teach SOLID, but not true in the context of an enterprise application with hundreds or thousands of classes.

Likewise, SOLID UI components will seem over-engineered if you only look at the trivial examples used to teach the practice. However, the correct setting to consider is an enterprise which has a UI builder, widgets, and a platform for users and vendors to add their own widgets. In that context, reuse and maintainability are key features.

Components in Go

The Go programming language has a very powerful built-in template language. It has a specialization for HTML generation. This system has been used to create extremely fast static web site generators like Hugo, used to generate this blog. Hugo also has an excellent template tutorial. If you already know go, another good place to learn about the template support for static web sites is the blog post by Alex Edwards here.

The closest thing to components in a statically generated web site are the templates used to convert data into HTML.

A component is a named HTML template with named parameters. These will be supplied as keyed values in a parameter map at run time. Each of the values can itself be a parameter map in order to support passing parameters to nested components.

Values can be hard coded in the template, or derived from context that is injected into the template.

Component evolution is supported through composition and overrides.

How is this approach SOLID?

  • SRP: The ‘AboutOurTeam’ view model has a single responsibility. Composition and binding allow the templates themselves to remain simple. Here the focus is on the headings, message, and the list of team members.

  • OCP: Extending a template can be done by means of the [Shadow DOM][shadow], another HTML 5 feature. This allows an element’s content to be replaced by completely encapsulated HTML markup that has access to the original element.

    Aside. Suppose you created a widget that contained an h2, a p and a text input field to allow users to rate the content (the text can be * to *****). Now you wanted to replace the input field with a star rater. It is possible to shadow the original widget, keeping the h2 and p visible while displaying a star rater widget whose value feeds the invisible text field.

  • LSP: Careful delegation to the base of an extended component, as in the above example, ensures that the original contract is honored. The component interacts with the user in a different way - its view-facing interface has changed - but it will still be plug-in compatible with the original widget component.

  • ISP: Models are JavaScript objects. They can be decomposed into arrays, maps, or primitive values (string, number, or boolean). Loops decompose arrays. Name bindings decompose maps. Finally, primitives can have only a finite set of semantic types (e.g. number can represent an integer, decimal, money, percentage, distance, time, etc). These few semantic types make it possible to support a finite set of primitive interfaces for data binding.

  • DIP: The ‘AboutOurTeam’ view model had a dependency on the profile view model. This dependency may resolve to a concrete type (profile.html and profile.js in aurelia). But you can imagine a system where additional logic was involved in selecting a view model using additional information about the data instance to be bound and also the context. In particular resolution can take into consideration:

    • Context in the view
    • The type of data being bound (e.g. an extension of profile)
    • User based configuration
